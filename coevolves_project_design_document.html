1. 需求定义

项目不是从“我要写 SQL”开始，而是从监管报表和审计痛点开始。先把业务问题定义清楚，后面的技术选择才有方向。

40% 平台覆盖影响

PDF 提到 Canadian direct written premium managed through platform，说明数据规模和可信度要求高。

99.9% 可靠性目标

SaaS 平台需要稳定运行，监管数据链路不能依赖个人电脑和临时查询。

2 weeks 快速 onboarding

新 carrier 或新数据源接入后，验证规则必须可复用，不能每次从零手工检查。

hours 审计准备目标

目标是把 weeks of audit prep 变成 hours，通过自动测试、失败行和运行记录完成。

Functional requirements

  • 接入 carrier、broker、policy、premium transaction 数据。
  • 生成 regulatory premium mart。
  • 自动验证关键字段、重复交易、金额核对、数据新鲜度。
  • 向 dashboard 和 audit package 输出可信数据。

Non-functional requirements

  • Cloud native，不依赖本地手动流程。
  • 支持 role-based access control。
  • 所有规则和变更可追踪。
  • 失败时有日志、告警和修复路径。

Out of scope

  • 不重建完整 policy administration system。
  • 不直接替代 regulator submission portal。
  • 不在 dashboard 里临时修正数据。
  • 不把业务规则只保存在个人 SQL 文件。

3. 系统架构设计

这一层回答“具体系统怎么分层”。SaaS 项目落地时，前台产品、数据平台、质量控制和报表输出必须解耦。

Upstream systems

  • Policy administration systems
  • Broker / agent activity feeds
  • CRM and third-party providers
  • Premium ledger and payment systems
  • Regulatory mapping tables

Regulatory Data Quality Service

  • Ingest raw policy and premium data
  • Transform through dbt staging, intermediate, mart layers
  • Run SQL validation tests as quality gates
  • Publish trusted regulatory mart and audit artifacts
  • Alert owners when quality gate fails

Downstream outputs

  • Regul8 regulatory reporting
  • Tableau operational dashboards
  • Exception management reports
  • Audit evidence package
  • Executive KPI summaries

为什么用 dbt

监管报表逻辑需要版本控制、依赖管理、重复运行、测试结果和文档。dbt 把 SQL 从个人查询变成团队可维护的数据产品。

为什么用 cloud warehouse

数据规模可能达到 millions of transactions，本地 Excel 或单机脚本不适合承载生产级监管报表链路。

为什么要 audit layer

审计不是只问“数字是多少”，还会问“你怎么知道它是对的”。audit layer 提供可追溯证据。

5. 数据质量门设计

这是第二个简历 bullet 的核心。所谓 SQL validation scripts，就是把下面这些质量门写成 dbt tests。

Gate 1 Freshness: carrier 数据必须在 SLA 内到达，否则 regulatory report 不能生成。
Gate 2 Completeness: policy_id、transaction_id、premium、province、reporting_month 不允许缺失。
Gate 3 Reconciliation: transaction-level premium sum 必须对上 ledger/report total。
Gate 4 Auditability: 每次运行保存结果、失败行、Git version 和修复状态。
Example dbt custom test tests/assert_premium_reconciles_to_ledger.sql with detail as ( select reporting_month, province, sum(written_premium) as detail_premium from {{ ref('fct_policy_premium_transactions') }} group by 1, 2 ), ledger as ( select reporting_month, province, sum(ledger_premium) as ledger_premium from {{ ref('fct_premium_ledger') }} group by 1, 2 ) select d.reporting_month, d.province, d.detail_premium, l.ledger_premium, d.detail_premium - l.ledger_premium as difference from detail d join ledger l using (reporting_month, province) where abs(d.detail_premium - l.ledger_premium) > 1.00

6. 上线与运维设计

一个真正落地的 SaaS 数据项目，不能停在 notebook 或本地 SQL。它必须进入 Git、CI、scheduler、monitoring 和 support workflow。

Development

Data Analyst 在 VS Code 或 dbt Cloud IDE 里写 SQL models、YAML tests 和 custom SQL tests。用开发 schema 跑 `dbt build --select tag:regulatory_report`，确认模型和测试能跑通。

$ dbt build --select tag:regulatory_report PASS source_freshness_policy_transactions PASS not_null_policy_id PASS unique_transaction_id PASS assert_premium_reconciles_to_ledger

7. 实施路线图

如果这个项目真的要落地，可以按四个阶段推进。这样面试时你能讲清楚“先做什么，后做什么”。

Phase 1

Discovery and data profiling

梳理监管报表字段、数据源、业务口径、历史审计问题。对 raw policy transaction 和 premium ledger 做 profiling。

Deliverable: data dictionary + validation rule inventory
Phase 2

dbt modeling foundation

建立 staging、intermediate、regulatory marts。统一字段命名、数据类型、日期窗口和金额符号。

Deliverable: dbt models + docs site
Phase 3

Automated validation gates

实现 generic tests、自定义 SQL reconciliation tests、freshness checks、exception exports。

Deliverable: quality gate job + failed row tables
Phase 4

Reporting and audit readiness

把通过验证的数据接入 Tableau/reporting，并生成 audit pack：规则清单、运行结果、异常修复记录。

Deliverable: dashboard + audit evidence package

8. 风险、缓解方案和验收标准

Design document 不能只画漂亮流程，还要说明项目哪里可能失败，以及怎样判断它真的完成了。

Risk Why it matters Mitigation Acceptance criteria
上游数据延迟 报表可能基于过期数据。 source freshness test + SLA alert。 超过 SLA 的 source 会自动 warn/fail，不会静默进入报表。
监管口径变化 旧规则可能不再合法。 规则版本化，mapping table 加 effective date。 可以按 reporting period 重现当时的规则。
测试误报太多 团队会忽略告警。 设置 severity、tolerance、issue owner。 critical tests fail，non-critical tests warn，owner 明确。
SQL 性能慢 审计前运行时间过长。 增量模型、分区过滤、先聚合再 join。 regulatory validation job 在约定窗口内完成。
缺少审计证据 即使数据正确，也难以证明。 保存 run results、failed rows、Git SHA、修复记录。 每次报表生成都有可导出的 audit pack。

9. 面试时的项目叙述

这个 design document 最后要服务你的简历。下面这段可以把“设计”和“落地”一起讲清楚。

English version

I worked on a regulatory reporting data quality workflow for a P&C insurance SaaS platform. The goal was to make premium and policy transaction data reliable before it reached regulatory reporting and client dashboards. I helped design the data flow from ingestion into the warehouse, standardized the data through dbt models, and implemented SQL validation tests for completeness, duplicate transactions, premium reconciliation, business rules, and source freshness. These tests were version-controlled, reviewed through Git, and run in scheduled cloud jobs. The result was a repeatable audit-ready process with failed row outputs, logs, and evidence packages, reducing audit preparation from weeks to hours.

你要记住的落地点

  • 业务目标：监管报表准确、及时、可审计。
  • 产品位置：Regul8 regulatory reporting data platform。
  • 技术实现：cloud warehouse + dbt models + SQL tests + scheduler。
  • 你的贡献：把人工检查规则产品化，变成自动运行的 quality gates。
  • 业务影响：更快审计准备、更少人工错误、更清楚的数据责任归属。
