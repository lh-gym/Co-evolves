把技术先放慢一点

REST API 就像一个标准化的点餐窗口

你不用冲进厨房，也不用知道厨师怎么做菜。你只要按菜单说清楚“我要什么”，窗口就会把结果给你。软件之间也是这样：前端、手机 App、网页、后台服务，通过 REST API 用统一规则互相沟通。

  • 1你是客户端：网页、App、Excel 插件、Python 脚本都可以是客户端。
  • 2API 是窗口：它规定你能问什么、怎么问、会得到什么格式的回答。
  • 3服务器是厨房：它真正读取数据库、计算结果、保存新数据。
你
客户端 网页或 App 发出请求：我想看订单、创建用户、修改资料。
→
API
点餐窗口 检查请求格式，决定把请求交给哪个后台功能。
→
库
服务器与数据库 真正处理数据，然后把结果交回给客户端。

REST API 的重点不是“高级代码”，而是“约定”。大家都按同一套规则说话，系统之间才不会鸡同鸭讲。

一句话定义

如果你只记一句，记这个就够了。

REST API 是一种让软件通过网址和 HTTP 动作来读取、创建、修改、删除资源的沟通规则。
GET https://api.shop.com/customers/42/orders
GET 动作：我要读取
api.shop.com 服务器地址
/customers/42 第 42 号客户
/orders 客户的订单资源

REST

一种设计风格，强调用清楚的网址表示资源，用固定 HTTP 动作表示操作。

API

Application Programming Interface，软件给其他软件使用的接口。

资源

系统里的一个东西，例如用户、订单、商品、文章、评论、模型预测结果。

REST API 最常见的四个动作

REST API 通常把“你想做什么”放进 HTTP 方法里。你可以把它们理解成数据库操作的日常语言版本。

GET
读取 查看已有数据。例如：查看所有客户、查看某个订单。
POST
创建 新增一条数据。例如：创建新客户、提交新订单。
PUT
更新 修改已有数据。例如：更新客户电话、修改订单状态。
DELETE
删除 移除一条数据。例如：删除草稿、取消一条记录。

点击左边不同动作，观察请求、解释和响应如何变化。这里用“任务清单”做例子，因为任务就是一种资源。

一次 API 请求背后发生了什么

当你在网页上点一个按钮时，看起来只是页面变化了。实际背后通常经历了下面这些步骤。

1 用户触发动作 比如点击“查看订单”或“保存资料”。
2 客户端发请求 带上方法、网址、必要参数和身份信息。
3 API 接收请求 检查你有没有权限、格式对不对、资源存不存在。
4 服务器处理数据 查询数据库、计算结果、创建或修改记录。
5 返回响应 给出状态码和 JSON 数据，客户端再展示给用户。

状态码就是服务器的简短回复

REST API 不只返回数据，还会返回状态码。状态码让客户端快速知道“这次请求成功了吗，如果失败，为什么失败”。

200

成功

请求正常完成，常见于 GET 或 PUT。

201

已创建

新资源创建成功，常见于 POST。

400

请求有问题

参数格式错了，服务器看不懂。

401

未登录

需要身份验证，例如 token 过期。

404

找不到

请求的资源不存在，例如任务 999 不存在。

REST API 和普通网页有什么区别

很多初学者会问：既然都是网址，那 API 和网页是不是一样？它们都通过网络访问，但目的不同。

普通网页 REST API
给人看的页面。 给软件读的数据。
返回 HTML、CSS、图片。 常返回 JSON。
重点是排版和视觉。 重点是结构清楚、字段稳定。
例子：打开购物网站首页。 例子：请求某个商品的价格和库存。

一个真实感很强的例子

你在手机 App 点“查看我的订单”。App 不会把整个数据库下载下来，也不会直接进入公司后台数据库。它会请求一个接口，比如 GET /api/orders?customer_id=42。 服务器检查你是谁，然后只返回你的订单数据。App 拿到 JSON 后，再把它排版成你看到的订单列表。

最后用四句话记住 REST API

第一次理解 REST API，不需要把所有技术细节背下来。先把这四句话吃透。

API 是窗口 客户端通过窗口请求数据，不直接碰后台数据库。
URL 找资源 例如 `/tasks/1` 表示第 1 个任务。
方法表动作 GET 读，POST 增，PUT 改，DELETE 删。
JSON 传结果 服务器通常用 JSON 把结构化数据交回给客户端。
